# Codebase Audit — stephan.zych.be

**Date:** 2026-06-09
**Scope:** `src/**` (TS app code, `.njk` templates, CSS), `.eleventy.js`, `esbuild.config.mjs`. Excluded: content prose (`*.md`), `_references/`, `_site/`, `node_modules/`.
**Source files analyzed:** ~109 (61 TS, ~5,300 LOC) + templates/config

## Executive Summary

- **Total findings:** 17 (0 critical, 6 warning, 11 info)
- **Top 3 areas needing attention:**
  1. **Untyped cross-module APIs** — several `as any` / `as unknown as` casts replace real interface contracts (slideshow, window↔window-manager, router payloads, search index).
  2. **Dead code** — one orphan/stale data file, two unused controllers + their barrel, and three unused/never-consumed action constants.
  3. **A couple of large, multi-responsibility modules** (`sz-window-manager.ts`, `sz-palette.ts`) with duplicated internal logic.
- **Overall assessment:** Genuinely healthy, well-layered architecture (no circular deps, no layering violations, clean core→feature→wiring separation). Findings are maintainability polish, not defects. The agents' theoretical perf "criticals" (100-item palettes, 1000-line gutters, many simultaneous windows) don't occur at this site's real scale and were down-rated accordingly.

## Dead Code

### Warning
- `src/controllers/search-filter.ts`, `src/controllers/line-gutter.ts`, `src/controllers/index.ts` — `SearchFilterController` and `LineGutterController` are defined and re-exported, but the barrel has **no importers** and neither class is instantiated anywhere (only `TypewriterController` is used, imported directly). Dead modules masquerading as public API. → delete all three (and keep `TypewriterController` imported directly, as it already is).

### Info
- `src/features/tmux/actions.ts:4` — `TMUX_ACTION.PANE_RESIZE` defined, never dispatched or listened. → remove.
- `src/features/neovim/actions.ts:2` + `src/features/neovim/sz-palette.ts:162` — `NEOVIM_ACTION.PALETTE_OPEN` has a registered listener but is **never dispatched**; the palette only opens via keydown. → either remove the action + listener, or wire a `:`/`/` trigger to dispatch it (decide intent).
- `src/features/window-manager/actions.ts:9` — `WM_ACTION.MODE_CHANGED` is dispatched 3× (`sz-window-manager.ts:469/484/491`) but **nothing listens**. Harmless but pointless. → remove dispatches, or add the intended consumer (e.g. statusbar mode indicator).

## Duplication

### Warning
- `src/data/commands.json` ↔ `src/app/wiring/neovim-palette.ts:18-41` — the JSON duplicates the command list that is actually hardcoded in the palette wiring, and it has **already drifted** (its `set` args still read `["transparency","shader"]` without the value lists added in code). It is never loaded. → delete `commands.json` (single source of truth = the wiring), or invert it (load the JSON, drop the hardcode). Recommend delete.

### Info
- `src/features/effects/sz-effect-matrix.ts` & `sz-effect-confetti.ts` — identical `cleanupEffect?: () => void` lifecycle boilerplate (~15 lines each). → optional shared base class/mixin if a third effect appears.
- `src/features/neovim/sz-palette.ts:444-475` — `executeItem`, `executeResolvedCommand`, `executeFromInput` repeat "fetch base items → find command → execute with args". → consolidate into one `executeCommand(label, args)`.
- `src/features/neovim/sz-palette.ts:359-379` — `currentFragment`, `confirmedTokens`, `ghostHint` each re-split `this.input`. → one `parseInput()` returning `{tokens, fragment, confirmed}`.

## Interface Clarity

### Warning
- `src/app/wiring/slideshow.ts:6,11,12` — slideshow discovered via `querySelector(...) as any` and called with `setImages/next/prev` untyped. Renaming a method breaks this silently at runtime. → export a `SlideshowApi` interface from `components/sz-slideshow.ts`, type the query.
- `src/features/neovim/sz-statusbar.ts:17` & `src/features/tmux/sz-tmux-bar.ts:18` — both do `(a.payload as any).path` on the router's `ROUTE_CHANGED`. → export a `RouteChangedDetail` type from `core/router.ts` (or `core/actions.ts`) and use it in both.
- `src/app/wiring/neovim-palette.ts:74-76` — search index fetched then cast `as any[]` / `entry: any`, even though `registry.getSearchIndex()` already returns a typed `SearchEntry[]`. → drop the casts, use the existing type.

### Info
- `src/features/window-manager/sz-window-manager.ts:536` — `w(el) { return el as unknown as WindowApi }` used ~40×. The cast is well-isolated but signals a missing real contract (see Coupling W).

## Coupling

### Warning
- `src/features/window-manager/sz-window-manager.ts:1-30,536` ↔ `src/features/window/sz-window.ts:218-291` — `WindowApi` is declared **privately inside** the manager and applied to `sz-window` via an unchecked cast. The two modules share a large implicit contract (`setLayout`, `getLayout`, `bringToFront`, `showWindow`, `enterFullscreen`, …) with no compile-time link. → move `WindowApi` to a shared spot (`window/index.ts` or `core/types.ts`) and have `SzWindow implements WindowApi`. Kills the `as unknown as` bridge and gives both sides type safety.

### Info
- `src/core/registry.ts` — centralizes nav, socials, wallpapers, shortcuts, search index, and site metadata; consumed by 5+ modules. Acceptable as a small site's config hub; flagged only so it doesn't accrete further. Splitting (site-data vs UI-data) is optional.

## File Organization

### Warning
- `src/features/window-manager/sz-window-manager.ts` (551 lines) — one component owns drag, resize, tiling, maximize, fullscreen, z-stack, mobile, and focus. Hard to test a single concern in isolation. → (boring, larger refactor) extract drag/resize/tiling engines. Defer unless it keeps growing.
- `src/features/*/index.ts` — inconsistent barrels: most re-export only the component class, not the feature's `actions.ts`, so consumers import deep paths (`../tmux/actions.js`). → standardize each `index.ts` to also `export * from './actions.js'`.

### Info
- `src/layouts/` mixes `.njk` templates with `.ts` Lit components. Intentional and small, but a brief note in the README/structure docs would remove ambiguity.
- `src/components/slideshow-actions.ts` lives outside a feature folder and doesn't follow the co-location pattern used elsewhere. Minor.

## Performance & Optimization

### Info
- `src/layouts/sz-markdown.ts:138-172` — `updateLineNumbers()` runs on every `ResizeObserver` callback and image load, re-reading `getComputedStyle` and rebuilding the gutter via `innerHTML`, with no height dirty-check/debounce. Negligible at current content sizes; cheap win = bail early when measured height is unchanged. → low priority.
- `src/features/neovim/sz-palette.ts:383` — `loadItems()` runs synchronously on every keystroke. Fine for the in-memory command source; matters only if a slow async source is added. → debounce if/when that happens.
- `src/features/effects/sz-effect-matrix.ts:74-81` — rapid re-trigger could overlap an interval before cleanup. Edge case; ensure prior cleanup runs at the top of `start*()`.

## Portability & Reusability

### Info
- Feature components (`sz-tmux-bar`, `sz-palette`, `sz-statusbar`, `sz-start-screen`, `sz-dashboard`) read the global `registry`/`appState` singletons directly. The earlier agents flagged this as "critical non-portability," but this is an **application**, not a component library — singleton config is a reasonable choice here. Listed for awareness only; not an action item unless extraction to a library becomes a goal.
